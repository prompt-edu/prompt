# PROMPT 2.0 — `prompt-websec` test-prod deployment

A self-contained, **HTTPS** PROMPT deployment for the IT-security course, reachable at
**https://prompt-websec.aet.cit.tum.de/**. It bundles an embedded
Keycloak (with seeded example users) so the whole stack runs from one Docker Compose project.

> This deviates from the stock production setup on purpose — see "Design notes" below. Do **not**
> use this configuration for a real production deployment.

## Files

| File | Committed | Purpose |
|---|---|---|
| `docker-compose.websec.yml` | yes | Custom compose: TLS-terminating Traefik, embedded Keycloak under `/idp`, all services |
| `.env.websec.example` | yes | Env template with `__GENERATE__` placeholder secrets |
| `keycloakConfig.websec.json` | yes | Realm import (example users) with `__PROMPT_SERVER_SECRET__` / `__ADMIN_PASSWORD__` placeholders |
| `render-secrets.sh` | yes | Generates `.env.websec` + `keycloakConfig.rendered.json` with real secrets |
| `privacy.html`, `imprint.html`, `chair-logo.jpeg` | yes | Placeholder assets mounted into the core frontend |
| `.env.websec` | **no (gitignored)** | Real secrets, generated on the VM |
| `keycloakConfig.rendered.json` | **no (gitignored)** | Realm import with the real `prompt-server` secret |

The deployment also starts the Intro Course and GitHub Challenge external micro-frontends. They are
served at both their legacy paths (`/intro-course`, `/devops-challenge`) and the current core paths
(`/intro-course-developer`, `/github-challenge`) so Module Federation remotes resolve on this host.

## Deploy from scratch (on the VM, as the `prompt` user)

```bash
# 1. Code
git clone https://github.com/prompt-edu/prompt.git ~/prompt
cd ~/prompt
git checkout deploy/prompt-websec

# 2. Docker network (once)
docker network create prompt-network   # ignore "already exists"

# 3. Secrets (writes deploy/.env.websec and deploy/keycloakConfig.rendered.json)
bash deploy/render-secrets.sh

# 4. Launch
docker compose -f deploy/docker-compose.websec.yml --env-file deploy/.env.websec up --pull=always -d

# 5. Watch the slow starters
docker compose -f deploy/docker-compose.websec.yml --env-file deploy/.env.websec logs -f keycloak server-core
```

First Keycloak boot builds the server and imports the realm (~1–2 min); backends run their DB
migrations on start. Once `keycloak` and `server-core` are healthy the site is up.

## Seeded users (realm `prompt`)

| Username | Password | Role |
|---|---|---|
| `admin` | generated (= `KEYCLOAK_ADMIN_PASSWORD`) | PROMPT_Admin |
| `lecturer` | `lecturer` | PROMPT_Lecturer |
| `course-lecturer` | `course-lecturer` | course lecturer |
| `course-editor` | `course-editor` | course editor |
| `student` | `student` | student |
| `student-passkey` | `student-passkey` | student |

The privileged `admin` account does **not** ship with a weak password: `render-secrets.sh` injects
the generated `KEYCLOAK_ADMIN_PASSWORD` (read it from `deploy/.env.websec`). The remaining accounts
are deliberately simple role-test logins for the course, matching the stock realm seed.

Keycloak admin console: `https://prompt-websec.aet.cit.tum.de/idp/admin/` (bootstrap-admin
credentials are in `deploy/.env.websec`, generated by `render-secrets.sh`).

## Common operations

```bash
C="docker compose -f deploy/docker-compose.websec.yml --env-file deploy/.env.websec"
$C ps                 # status
$C logs -f server-core
$C up --pull=always -d # update to latest :main images
$C down               # stop (keeps volumes/data under deploy/)
$C down -v            # stop and DELETE all data volumes
```

## Design notes (why this differs from `docker-compose.prod.yml`)

- **HTTPS with automatic certificates.** Traefik terminates TLS on `:443`, redirects `:80` to HTTPS,
  and obtains/renews a Let's Encrypt certificate via the TLS-ALPN challenge (prod uses the HTTP-01
  challenge; both require ports 80/443 to be publicly reachable). Certificate state persists in
  `deploy/letsencrypt/`.
- **Embedded Keycloak under `/idp`.** Prod points at an external Keycloak; here it is bundled. Only
  one hostname resolves (no DNS subdomains), so Keycloak runs under the `/idp` relative path
  (`KC_HTTP_RELATIVE_PATH=/idp`). Both the frontend and backend build Keycloak URLs as
  `${KEYCLOAK_HOST}/realms/<realm>`, so a path-bearing `KEYCLOAK_HOST` works.
- **`CORE_HOST` carries the scheme.** The Go backend uses `CORE_HOST` as a full URL (CORS, mail
  links, inter-service base URLs), so it is `https://prompt-websec.aet.cit.tum.de`. Traefik `Host()`
  rules therefore use the **bare** domain literally instead of `${CORE_HOST}`.
- **Keycloak network alias.** The reverse-proxy container carries the network alias
  `prompt-websec.aet.cit.tum.de`, so backends resolve the public FQDN to Traefik *inside* the Docker
  network. OIDC issuer validation then reaches Keycloak at the exact public URL it advertises (the
  issuer must match byte-for-byte) without leaving the host.

## Known limitations

- **File uploads (S3):** presigned URLs are signed against `/s3`, which path-style signing across a
  Traefik prefix may reject. Internal storage works; browser presigned upload/download is untested.
- **Email:** no working SMTP on the VM; mail-sending features will no-op.
- **Memory:** ~24 containers on ~3.9 GB RAM + 2 GB swap — tight. Watch `docker stats` / `free -m`.
