# Infrastructure Setup Phase

## Overview

The **Infrastructure Setup Phase** is a course phase that automates the creation of external resources (GitLab groups, Slack channels, Outline collections, Rancher projects, Keycloak groups) per team or student. It eliminates the manual, error-prone process of setting up infrastructure for large courses.

Given teams from a preceding Team Allocation phase, the service creates and configures external resources based on instructor-defined templates and permission mappings.

## Five Providers

### 1. GitLab

| | |
|---|---|
| Auth | `base_url`, `private_token`, `parent_group_id` (optional) |
| Resource type | `group` |
| Permission levels | Guest (10), Reporter (20), Developer (30), Maintainer (40), Owner (50) |

**API calls:**
- `GET /api/v4/groups?search=<name>` - idempotency check (compare `full_path`)
- `POST /api/v4/groups` - create group
- `GET /api/v4/users?search=<email>` - look up member by email
- `POST /api/v4/groups/:id/members` - add member (HTTP 409 = already member = success)
- `DELETE /api/v4/groups/:id` - delete group
- `GET /api/v4/version` - validate credentials

### 2. Slack

| | |
|---|---|
| Auth | `bot_token` (xoxb-...) with scopes: `channels:manage`, `channels:write.invites`, `groups:write`, `users:read`, `users:read.email` |
| Resource type | `channel` |
| Permission levels | member, owner |

**API calls:**
- `POST /conversations.create` - create channel (default `is_private: true`)
  - Handle `already_name_taken` by looking up existing channel (`conversations.list`)
- `POST /users.lookupByEmail` - resolve email to Slack user ID
- `POST /conversations.invite` - add members (handle `already_in_channel` as non-error)
- `POST /conversations.archive` - archive channel on delete (bot must be channel creator or workspace admin)
- `POST /auth.test` - validate credentials

Channel name constraints: max 80 chars, lowercase, no spaces (slugify + truncate applied automatically).

### 3. Outline

| | |
|---|---|
| Auth | `api_key` (ol_api_...), `base_url` (default: https://app.getoutline.com/api) |
| Resource type | `collection` |
| Permission levels | read, read_write, admin |

**API calls:**
- `POST /collections.create` - create collection
- `POST /users.list` - find user by email (paginate + match on email field)
- `POST /memberships.create` - add user to collection (collectionId, userId, permission)
- `POST /collections.delete` - delete collection
- `POST /auth.info` - validate credentials

> **Important:** The correct endpoint for adding a member is `POST /memberships.create`, NOT `collections.add_user`.

### 4. Rancher

| | |
|---|---|
| Auth | `rancher_url`, `access_key`, `secret_key` (Basic Auth), `cluster_id` |
| Resource type | `project` |
| Permission levels | read-only, project-member, project-owner |

**API calls:**
- `GET /v3/projects?name=<name>&clusterId=<id>` - idempotency check
- `POST /v3/projects` - create project (clusterId required)
- `GET /v3/users?email=<email>` - look up user
- `POST /v3/projectroletemplatebindings` - add member (projectId, userPrincipalId, roleTemplateId)
- `DELETE /v3/projects/:id` - delete project
- `GET /v3` - validate credentials

> **Important:** `roleTemplateId` (e.g., `project-member`) is a required field stored in `resource_config`, not in credentials.

### 5. Keycloak

| | |
|---|---|
| Auth | `keycloak_url`, `realm`, `client_id`, `client_secret` |
| Resource type | `group` |
| Permission levels | member (only one level) |

**API calls:**
- `POST /realms/{realm}/protocol/openid-connect/token` - get client_credentials token (cached, refreshed before expiry)
- `GET /admin/realms/{realm}/groups?search=<name>` - idempotency check (compare exact `name`)
- `POST /admin/realms/{realm}/groups` - create group
- `GET /admin/realms/{realm}/users?email=<email>` - look up user (handle empty array)
- `PUT /admin/realms/{realm}/users/{userId}/groups/{groupId}` - add user to group
- `DELETE /admin/realms/{realm}/groups/{groupId}` - delete group
- `GET /admin/realms/{realm}` - validate credentials

> **Security:** The Keycloak provider uses the same Keycloak instance PROMPT uses for authentication. The service account client must be granted **only** `manage-groups` and `view-users` realm roles — **NOT** `realm-admin`. This limits the blast radius if credentials are compromised.

## Database Schema

```sql
CREATE TYPE provider_type AS ENUM ('gitlab', 'slack', 'outline', 'rancher', 'keycloak');
CREATE TYPE resource_scope AS ENUM ('per_team', 'per_student');
CREATE TYPE resource_status AS ENUM ('pending', 'in_progress', 'created', 'failed');

-- Stores encrypted provider credentials, one row per (phase, provider type)
CREATE TABLE provider_config (
    id               uuid PRIMARY KEY,
    course_phase_id  uuid NOT NULL,
    provider_type    provider_type NOT NULL,
    credentials      bytea NOT NULL DEFAULT ''::bytea,  -- AES-256-GCM: 12-byte nonce || ciphertext
    CONSTRAINT uq_provider_config_phase_type UNIQUE (course_phase_id, provider_type)
);

-- Defines what resources to create, with name templates and permission mappings
CREATE TABLE resource_config (
    id                 uuid PRIMARY KEY,
    course_phase_id    uuid NOT NULL,
    provider_type      provider_type NOT NULL,
    resource_type      text NOT NULL,
    scope              resource_scope NOT NULL,
    name_template      text NOT NULL,  -- supports {{teamName}}, {{semesterTag}}, {{studentFirstName}}, {{studentLastName}}, {{studentEmail}}
    permission_mapping jsonb NOT NULL DEFAULT '{}',  -- {"student": "developer", "tutor": "maintainer", "instructor": "owner"}
    created_at         timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_resource_config_provider
        FOREIGN KEY (course_phase_id, provider_type)
        REFERENCES provider_config (course_phase_id, provider_type) ON DELETE CASCADE
);

-- Tracks each provisioned resource instance with its status
CREATE TABLE resource_instance (
    id                       uuid PRIMARY KEY,
    resource_config_id       uuid NOT NULL REFERENCES resource_config(id) ON DELETE CASCADE,
    course_phase_id          uuid NOT NULL,
    team_id                  uuid,
    course_participation_id  uuid,
    status                   resource_status NOT NULL DEFAULT 'pending',
    external_id              text,
    external_url             text,
    error_message            text,
    retry_count              integer NOT NULL DEFAULT 0,
    created_at               timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at               timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Prevent duplicate non-failed instances for the same (config, team/student)
CREATE UNIQUE INDEX uq_resource_instance_team
    ON resource_instance (resource_config_id, team_id)
    WHERE team_id IS NOT NULL AND status != 'failed';

CREATE UNIQUE INDEX uq_resource_instance_student
    ON resource_instance (resource_config_id, course_participation_id)
    WHERE course_participation_id IS NOT NULL AND status != 'failed';
```

## Credentials Encryption

Credentials are stored as AES-256-GCM encrypted `bytea` in the database.

**Format:** `12-byte nonce || ciphertext` (concatenated)

**Key:** Environment variable `ENCRYPTION_KEY` (base64-encoded 32-byte key)

**Behavior:**
- A new random 12-byte nonce is generated per encryption call using `crypto/rand`
- The API **never** returns decrypted credentials; only `isConfigured: bool` is exposed
- Key rotation is a manual ops procedure: re-encrypt all `provider_config.credentials` rows

**Generate a key:**
```bash
openssl rand -base64 32
```

## Name Templates

Resource names support the following template variables:

| Variable | Description |
|---|---|
| `{{teamName}}` | Name of the team |
| `{{semesterTag}}` | Course semester tag (e.g., `ss25`) |
| `{{studentFirstName}}` | Student's first name (per_student scope only) |
| `{{studentLastName}}` | Student's last name (per_student scope only) |
| `{{studentEmail}}` | Student's email address (per_student scope only) |

Template values are sanitized per-provider before use in API calls (e.g., Slack channel names are slugified and truncated to 80 chars).

## Async Execution Engine

1. `POST /execution/trigger` fetches teams + students from the PROMPT core API, creates `pending` instances for each `resource_config × target`, commits the DB transaction, then launches goroutines.
2. A goroutine pool (semaphore, max 5 concurrent) processes all pending instances.
3. Per instance: `in_progress` → resolve name template → call `provider.CreateResource()` → `created` + URL, or `failed` + error message.
4. On HTTP 429 or transient error: exponential backoff (1s, 2s, 4s + jitter), max 3 retries, then `failed`.
5. `retry_count` is tracked in the DB.

**Startup recovery:** On service start, any instance with `status = 'in_progress'` is reset to `'pending'` (covers server restart mid-execution).

**Idempotency:** `POST /trigger` returns 409 if non-terminal (`pending`/`in_progress`) instances already exist for the phase.

---

## Delete Semantics

When `DELETE /execution/instances/:id` is called:
- The service attempts `provider.DeleteResource()` to revoke the external resource
- **On provider API failure:** the instance is marked `failed` with the error message stored in `error_message`. The DB record is kept (audit trail). The response is 200 with error detail.
- **On success:** the instance record is deleted from the DB.

---

## Phase Copy Behavior

When a course phase is copied (e.g., next semester):
- `provider_config` rows are copied with **empty credentials** (the instructor must re-configure credentials)
- `resource_config` rows (name templates, permission mappings) are copied
- `resource_instance` rows are **not** copied (provisioning must be re-triggered)

## API Reference

All endpoints are under `/infrastructure-setup/api/course_phase/:coursePhaseID` and require `CourseLecturer` or `PromptAdmin` role.

| Method | Path | Description |
|---|---|---|
| GET | `/provider` | List all providers with auth field specs and `isConfigured` flag |
| PUT | `/provider/:providerType` | Upsert credentials (validates before storing) |
| POST | `/provider/:providerType/validate` | Test credentials live without storing |
| GET | `/resource-config` | List resource configs |
| POST | `/resource-config` | Create resource config |
| PUT | `/resource-config/:id` | Update resource config |
| DELETE | `/resource-config/:id` | Delete resource config |
| GET | `/execution/preview` | Count what instances would be created |
| POST | `/execution/trigger` | Create pending instances + start async execution (202) |
| GET | `/execution/instances` | List instances with status |
| POST | `/execution/instances/:id/retry` | Re-queue a failed instance |
| DELETE | `/execution/instances/:id` | Attempt to revoke external resource, mark failed on error |
| GET | `/config` | Readiness flags |
| POST | `/copy` | Phase copy handler |

## Test Coverage

### Unit Tests (no DB, mock HTTP)
- Encryption round-trip, wrong key returns error, nonce uniqueness (1000 calls), corrupted ciphertext error
- Template resolver: all 5 variables, special character sanitization, Slack 80-char truncation
- Each provider: create success, idempotency, member add edge cases (409, already_in_channel), delete, validate credentials, error handling
- Worker pool: max 5 concurrent, retry backoff, 3 retries then failed, context cancellation

### Integration Tests (testcontainers-go + real PostgreSQL)
- providerconfig: encrypt on upsert, GET returns only `isConfigured`, credentials never in response
- resourceconfig: CRUD, FK constraint rejects missing provider
- execution: trigger creates pending instances, 409 on duplicate trigger, delete-on-fail keeps record, retry resets to pending
- Startup recovery: manually set `in_progress`, call recovery, verify reset to `pending`
- copy: configs copied, credentials empty, instances absent

### Provider API Tests (httptest.NewServer mock)
- Per provider: create (success + idempotent), delete, validate, member lookup edge cases
